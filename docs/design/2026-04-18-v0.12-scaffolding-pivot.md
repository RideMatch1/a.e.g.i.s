## 1. Summary

v0.12 is AEGIS's first scope-pivot release. Where v0.9 в†’ v0.11 focused
entirely on scanner precision and recall, v0.12 adds two project-
lifecycle commands to the CLI:

- `aegis new <name>` вЂ” scaffold a new Next.js+Supabase SaaS with AEGIS
  security primitives, PR-gate, and AI-safety rules pre-wired from
  commit 0.
- `aegis init` вЂ” retrofit the same security skeleton onto an existing
  Next.js+Supabase project without touching its application code.

The template is intentionally **minimal security foundation**, not a
feature-rich starter. Success is defined by `aegis scan .` returning
`1000/A/0` on the freshly-scaffolded project and continuing to return
`1000/A/0` unmodified across 3 consecutive AEGIS releases (v0.13, v0.14,
v0.15).

## 2. Motivation

### 2.1 The drift problem that scaffolding solves

A developer starting a new Next.js+Supabase project reads the AEGIS
README and installs the CLI. They run `aegis scan` a week later once
they have 40+ API routes, and get 47 findings. Many are primitives
they could have copy-pasted from a reference implementation on day 0
(Zod-strict wrappers, `requireRole`, tenant-scoped query helpers).
Each retrofit-after-the-fact is a refactor with regression risk.

The post-v0.11.1 dogfood cycle proved this concretely on a real-world
Next.js+Supabase reference project: a mid-release scan found ~40 API
routes using non-strict Zod, drifting from the "Zod `.strict()` on ALL
API schemas" rule the project's own `CLAUDE.md` prescribed. Each
violation compounds until a brutal-audit forces correction.

Scaffolding with AEGIS-as-PR-gate from commit 0 inverts this: the
developer can't merge a route that lacks `requireRole` because the
gate blocks it. Discipline becomes mechanical, not memorised.

### 2.2 Retrofit as the volume use-case

Every solo-dev has more existing projects than new ones. A pivot that
only serves greenfield misses ~80% of addressable usage. `aegis init`
surfaces the same primitives behind a non-destructive command that
only writes new files (`.github/workflows/aegis.yml`, `aegis.config.json`,
optional `CLAUDE.md` if missing) вЂ” never modifies the user's
application code.

### 2.3 Why now

Prerequisites for scaffolding credibility:

- Scanner precision is high enough that a "1000/A baseline" claim holds
  (v0.11.2 closed the last known FP class from dogfood).
- Canary harness proves no silent-miss class across 4 phases (67/67).
- 8-project public corpus is stable (|О”|=0 across v0.11.2).
- Release discipline is proven: 13 consecutive releases with
  auto-built `.vsix`, 0 scrub incidents since v0.9.6.

The foundation is trustworthy enough to ship as a starter template.

## 3. Non-goals

- **No feature-rich starter.** No Stripe-pre-wired, no AI-chat-template,
  no i18n, no payments, no Kanban. These are domain-specific concerns
  that turn the template into an opinionated product starter rather
  than a security foundation.
- **No interactive module-picker.** `aegis new --tier=standard --with-stripe`
  is not v0.12. Combinatorial module composition (16 configs with
  4 optional modules, 256 with 8) is unmaintainable solo and defers
  user-demand data.
- **No multi-stack templates.** `templates/nextjs-supabase/` is the only
  template. SvelteKit, Remix, Next.js+Prisma come after `nextjs-supabase`
  has survived 3 AEGIS-version-updates without regression.
- **No branch-specific content.** Anything Physio/Wellness/Spa/Hotel/
  B2B/E-commerce-specific is deferred to `templates/<branch>/` in v0.14+
  pending demand data. Mixing branch-content into
  `nextjs-supabase/` kills general-purpose positioning.
- **No template-upgrade command in v0.12.** `aegis upgrade` that diffs
  a v0.12-scaffold against a v0.13 template is a legitimate future
  command but out of initial scope. v0.12 documents how to apply
  template-diffs manually; tooling follows demand.

## 4. Commands

### 4.1 `aegis new <name>`

Non-interactive by default. One required positional (`<name>`) +
optional flags that cover CI-driven usage.

```
aegis new my-saas
aegis new my-saas --stack=nextjs-supabase            # explicit (default)
aegis new my-saas --pm=pnpm                          # package manager (default: pnpm)
aegis new my-saas --skip-install                     # CI mode
aegis new my-saas --skip-git                         # CI mode
```

**Interactive prompts** вЂ” minimal by design. Two prompts max, only
shown when stdin is a TTY:

1. Package manager: `pnpm` / `npm` / `yarn` (auto-detected from user's
   global install, confirmed once).
2. Git-init confirmation: `Initialize git? [Y/n]` (defaults Y).

No prompts for "add Stripe", "add AI", "add i18n" вЂ” those are v0.13+
discussions informed by demand data, not v0.12 defaults.

**Flow:**

1. Create `<name>/` directory (error if exists and non-empty).
2. Copy `templates/nextjs-supabase/files/` contents to `<name>/`.
   Files ending in `.tpl` are rendered through a placeholder pass
   (`{{PROJECT_NAME}}`, `{{AEGIS_VERSION}}`) and written without the
   `.tpl` suffix. Non-`.tpl` files are copied byte-for-byte.
3. Run `pnpm install` (unless `--skip-install`).
4. Run `git init && git add -A && git commit -m "chore: initial commit
   from aegis@<version>"` (unless `--skip-git`).
5. Run `aegis scan .` вЂ” final output must be `1000/A/0`. If not,
   exit non-zero with a diagnostic.
6. Print next-steps: update `.env.local`, run Supabase migrations,
   open PR.

### 4.2 `aegis init`

Non-destructive retrofit. Adds AEGIS wiring to an existing project
without touching application code.

```
aegis init                          # interactive: confirm each addition
aegis init --ci                     # non-interactive, writes all
aegis init --force                  # overwrite existing aegis.config.json
```

**Scope:** writes these files if missing (skips if present unless `--force`):

- `.github/workflows/aegis.yml` вЂ” AEGIS PR-gate. The workflow pins
  `RideMatch1/a.e.g.i.s/ci/github-action@vX.Y.Z` to the CLI version
  the user ran (`aegis --version`), so the scaffold / retrofit is
  reproducible and the scanner running in CI matches the one the
  user has locally.
- `aegis.config.json` вЂ” pre-tuned defaults matching the template.
- `.husky/pre-push` вЂ” local scan (only if `.husky/` exists already вЂ”
  never installs Husky itself).
- `CLAUDE.md` вЂ” AI-safety rules (only if missing at project root).

**Explicitly out-of-scope for v0.12 init:**

- Detecting missing primitives in user's code (`"you don't have a
  tenant-guard вЂ” want me to add one?"`). That's a non-trivial analysis
  and could insert code that breaks existing patterns. User can copy
  the primitives from the scaffold themselves; tooling follows demand.
- Modifying user's `tsconfig.json`, `next.config.js`, or any
  application code.
- Installing new dependencies. User runs their own `pnpm install` after
  `aegis init`.

Sum: 9 + 5 + 3 + 1 + 1 + 1 + 1 + 1 = 22. Anything tempting outside
this list goes to "Non-goals" or "v0.13+".

## 6. Primitive source-strategy

Every primitive in the template belongs to exactly one column. The
policy is mechanical вЂ” no per-file judgement calls during
implementation.

| Primitive | Source | LoC | Rationale |
|---|---|---|---|
| `lib/security/safe-fetch.ts` | **Clean-room** | ~30 | Reference-impl from web standards; zero domain knowledge |
| `lib/security/crypto.ts` (timingSafeStringEqual) | **Clean-room** | ~10 | Stdlib primitive |
| `lib/security/crypto.ts` (AES-256-GCM) | **Clean-room** | ~40 | Standard-lib wrapper; published pattern |
| `lib/validation/input.ts` (escapePostgrestLike) | **Clean-room** | ~15 | Well-documented escape rules |
| `lib/validation/input.ts` (isValidUUID, sanitizeString) | **Clean-room** | ~20 | Standard regex / stdlib |
| `lib/security/rate-limit.ts` (getTrustedClientIp) | **Clean-room** | ~25 | Documented X-Forwarded-For logic |
| `lib/security/rate-limit.ts` (checkIPRateLimit) | **Clean-room** | ~40 | In-memory rate-limit pattern |
| `lib/api/require-role.ts` | **Clean-room** | ~30 | Simple RBAC shape, no hidden semantics |
| `lib/errors.ts` | **Clean-room** | ~40 | Standard error-class hierarchy |
| `.github/workflows/aegis.yml` | **Clean-room** | ~30 | Writes from `ci/github-action/action.yml` schema |
| `aegis.config.json` | **Clean-room** | ~20 | Config shape, no code inheritance |
| **Total clean-room** | **11 primitives** | **~300 LoC** | Zero scrub-risk, ~2вЂ“3h total impl |
| `lib/api/tenant-guard.ts` (secureApiRouteWithTenant) | **Extract+scrub** | ~80 | Complex flow; tenant-id extraction + auth + RLS-client-choice has real-world edge-cases |
| `lib/logger.ts` | **Extract+scrub** | ~150 | 50+ PII patterns are real-production-discovered; clean-room rewrite loses coverage |
| `middleware.ts` | **Extract+scrub** | ~80 | CSRF + headers + rate-limit coordination has ordering edge-cases |
| `supabase/migrations/0000_rls_bootstrap.sql` | **Extract+scrub** | ~60 | RLS-policies are battle-tested; wiedererfinden = recall-risk |
| `lib/validation/schemas.ts` (Zod-strict wrapper) | **Extract+scrub** | ~50 | API-schema shape is domain-evolved |
| `CLAUDE.md` | **Adapt from reference project** | ~150 lines | Structure is proven; contents are generalised (no domain-specific examples) |

| `app/api/example/[id]/route.ts` | **Clean-room composed** | ~40 | Exemplary route demonstrates all primitives together; written fresh to showcase composition |

**Directory structure** (`lib/api/`, `lib/security/`, `lib/validation/`):
copy convention from the reference project. Evolved over 160 commits;
re-inventing a layout from scratch risks ergonomic regressions.

### 6.1 Hard-scrub protocol (mandatory for every extract)

For each extracted primitive:

1. Copy file from the reference project to `templates/nextjs-supabase/files/<path>`.
2. Run the maintainer's **identity-scrub** grep (patterns kept in a
   maintainer-local file outside the public repo вЂ” they identify the
   source project and CANNOT appear in shipped tarballs) against the
   copied file вЂ” **zero hits required.**
3. Run the maintainer's **domain-scrub** grep (industry-specific terms
   from the reference project's branch вЂ” also kept maintainer-local)
   against comments, error messages, and variable names. Zero hits
   in code. Generic terms that survived outside the source branch
   are tolerated in prose comments only if they read naturally to a
   reader unaware of the source project.
4. Imports review: every `@/features/*` import must either be resolved
   (rewired to template paths) or removed.
5. Type-check isolation: primitive must compile standalone against the
   template's `tsconfig.json`. No dependency on reference-project-
   specific types.
6. AEGIS self-scan in an isolated project. Copy the primitive plus a
   minimal `tsconfig.json` into a tempdir, then run `aegis scan .`
   from that tempdir вЂ” require 0 findings. (AEGIS is a project-
   scanner, not a file-scanner; the tempdir is how we check a single
   primitive in isolation.)
7. Unit test: extract / adapt tests from the reference project, apply
   the same two scrub greps, run `pnpm test` вЂ” all green.

Every primitive that exits scrub gets a header comment:
`// Reference-implementation extract вЂ” generic Next.js+Supabase primitive.`
No further provenance. The maintainer's local scrub-term files are
the source of truth for step 2/3; they never land in the public repo
or in any shipped tarball.

## 7. Exit criteria

v0.12.0 ships when ALL of the following hold:

1. **Scaffold scores 1000/A/0.**
   ```
   aegis new test-proj && cd test-proj
   pnpm install
   aegis scan .
   # must output: score=1000, grade=A, findings=0
   ```

2. **Retrofit on a real existing project does not break it.**
   `aegis init` against a fresh clone of the maintainer's Next.js+
   Supabase reference project must complete without modifying any
   existing files beyond the 4 files in В§4.2. The project's
   `pnpm build` must still exit 0 after init.

3. **Full test suite covers scaffolding + retrofit.**
   - в‰Ґ 8 unit tests for `aegis new` (placeholder substitution,
     directory-exists error, skip-install, skip-git, post-scan baseline
     check, etc.)
   - в‰Ґ 6 unit tests for `aegis init` (no-overwrite, --force, missing
     `.husky`, etc.)
   - Regression canary: `templates/nextjs-supabase/` passes `aegis
     scan` as part of CI вЂ” a PR that breaks the template's baseline
     fails CI.

4. **Hard-scrub protocol verified on all 6 extracted primitives.**
   Scrub-log artifact committed showing zero hits per file per rule.

5. **Scaffold-regression job is part of AEGIS CI.** A recurring GitHub
   Action (on every AEGIS PR that touches `templates/`, `packages/scanners/`,
   or `packages/core/`) runs `aegis new ci-test-proj` in a tempdir,
   scans it, and fails the PR if the result is not `1000/A/0`. This is
   how the "3-release survival" claim gets enforced mechanically вЂ”
   every scanner change is automatically validated against the
   scaffold as a regression target. The 3-release stability is
   measured **post-v0.12.0** (success metric, В§8), not a pre-ship
   gate вЂ” v0.12.0 ships when the CI job exists and passes; v0.13
   and v0.14 will falsify or confirm the stability claim in the
   quarters that follow.

6. **Existing AEGIS gates unchanged:**
   - All 4 canary phases still green (67/67).
   - Self-scan 1000/A/0 on the AEGIS repo itself.
   - Scrub-list: 0 hits on tracked files + all 5 tarballs post-publish.
   - Honest score: 8.8 в†’ 8.9 (pivot is a capability addition).

## 8. Success metrics

### Post-ship (0вЂ“30 days)

- Qualitative: first external GitHub issue, discussion, or PR
  referencing a scaffolded-via-aegis-new project. `aegis new` ships
  with zero telemetry (matches the rest of the CLI's zero-telemetry
  posture), so adoption is only visible through public artifacts.
- Dogfood-CI: the `aegis new ci-test-proj` scaffold-regression job
  (exit criterion В§7.5) runs on every AEGIS PR touching `templates/`,
  `packages/scanners/`, or `packages/core/`. Its pass-rate is the
  primary "scaffold is healthy" signal.
- First community issue referencing the scaffold primitives (not the
  scanner). Signals adoption.

### Post-ship (30вЂ“90 days)

- One external project scanned that was scaffolded via `aegis new` в†’
  corpus growth. Tracks whether scaffolded projects actually stay
  compliant in real development.
- User-reported "missing primitive" issues. Informs `templates/` v2
  additions (data-driven, not speculative).

## 9. Open items (deferred to implementation plan)

- **Placeholder mechanism.** Handlebars-style `{{NAME}}` vs Python
  `.format` vs sentinel comments (`/* AEGIS_PLACEHOLDER: name */`).
  Decision in implementation plan; all three are simple and
  context-replacement-only.
- **`template.json` schema.** Minimal: `{ name, stack, aegisVersion,
  primitives: [...] }`. Schema finalised when first template is authored.
- **Version compatibility check.** `aegis new` with an older CLI
  version than the `templates/` aegisVersion should warn. Implementation-
  plan decision.
- **`.env.local.example`.** Always included, contains all required env
  vars with empty values. Finalised when Supabase migration shape is
  confirmed.

## 10. Open items (deferred to v0.13+)

- Module-composition prompts (`--with-stripe`, `--with-ai`, `--with-i18n`).
  Blocked on (a) 3-release stability proof for the base template,
  (b) real user-demand data on which modules are picked.
- Multi-stack templates (`templates/nextjs-prisma/`, `templates/sveltekit-supabase/`).
  Each new stack is a separate v0.13+ decision, gated on demand.
- `aegis upgrade` command to apply template-diffs to an existing
  scaffold-based project. Requires a diff-engine + user-opt-in flow.
- `aegis init` primitives-detection ("you're missing tenant-guard вЂ”
  add it?"). Requires non-trivial AST analysis and code-insertion
  safety. Not in v0.12.
- Branch-specific templates (`templates/physio-saas/`, `templates/b2b-saas/`).
  v0.14+, demand-gated.
